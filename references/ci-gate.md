# CI gate (`gen_ci.py`, #1)

Generated by `scripts/gen_ci.py --path <repo>`. Emits `.github/workflows/audit.yml`
and copies `scripts/` into the target repo as `.mincode/`.

## What the workflow does
1. `actions/checkout@v4` + `actions/setup-python@v5` (3.11).
2. `python .mincode/gen_tests.py .` — generate smoke tests.
3. `python -m unittest discover -s tests -v` — run them.
4. `python .mincode/audit.py .` — the gate. `audit.py` exits 1 if any HIGH
   finding, which fails the CI job. So a clean audit is *enforced*, not just
   tagged `audit-clean-<date>`.

## Gotchas (MUST follow or CI breaks)
- **Commit `.mincode/`** — it is NOT gitignored. The CI runner needs the
  scripts. If you gitignore it, the job fails with `No such file .mincode/...`.
- **Relative paths only.** The YAML uses `.mincode/audit.py`, never an absolute
  Windows path like `C:\Users\...\scripts`. Linux runners can't resolve that.
  `gen_ci.py` writes relative paths by design — don't "fix" it to absolute.
- **`unittest discover` can pass vacuously** (0 tests) in some envs. The audit
  gate (step 4) is the real enforcement; the test step is a smoke signal.
- After generating, commit **both** `.github/` and `.mincode/` and push — a PR
  with only one of them will fail.

## Manual run equivalent
```bash
python .mincode/gen_tests.py .
python -m unittest discover -s tests -v
python .mincode/audit.py .   # non-zero exit on HIGH -> fail
```
